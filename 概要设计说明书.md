## 1 引言

### 1.1 编写目的

在完成了校园美食店铺可视化推荐与评价小程序需求分析的基础上，我们依据《需求规格说明书》和《可行性分析报告》对项目进行了整体上的设计。本说明书旨在将项目的设计方案文档化，为后续详细设计和编码实现提供依据。

### 1.2 术语定义

| 术语 | 定义 |
|------|------|
| UGC | 用户生成内容（User Generated Content），指学生上传的店铺、菜品、评价等 |
| 随机推荐 | 小程序核心功能，随机匹配一家店铺推荐给用户 |
| 评分分布 | 展示店铺或菜品的星级评分比例分布 |
| 菜品互动 | 用户对菜品进行点赞、收藏、评论的行为 |
| 新店推荐 | 展示近期学生上传的新店铺 |
| 游客 | 访问小程序但还未注册或未进行登录的用户 |
| 用户 | 已通过微信授权登录的在校学生 |

### 3.2 数据库表之间的关系

数据库表之间的关系描述如下：

- **users 表**与其他表的关系：
  - 一个用户可以发表多条评价（1:N，reviews.user_id）
  - 一个用户可以上传多个店铺（1:N，shops.uploader_id）
  - 一个用户可以上传多个菜品（1:N，dishes.uploader_id）
  - 一个用户可以收藏多个店铺（1:N，favorite_shops.user_id）
  - 一个用户可以收藏多个菜品（1:N，favorite_dishes.user_id）
  - 一个用户可以点赞多个菜品（1:N，dish_likes.user_id）
  - 一个用户可以发表多条菜品评论（1:N，dish_comments.user_id）

- **shops 表**与其他表的关系：
  - 一个店铺属于一个类型（N:1，shops.type_id -> shop_types.id）
  - 一个店铺属于一个区域（N:1，shops.area_id -> areas.id）
  - 一个店铺可以有多条评价（1:N，reviews.shop_id）
  - 一个店铺可以有多个菜品（1:N，dishes.shop_id）
  - 一个店铺可以被多个用户收藏（1:N，favorite_shops.shop_id）

- **dishes 表**与其他表的关系：
  - 一个菜品属于一个店铺（N:1，dishes.shop_id -> shops.id）
  - 一个菜品可以被多个用户点赞（1:N，dish_likes.dish_id）
  - 一个菜品可以被多个用户收藏（1:N，favorite_dishes.dish_id）
  - 一个菜品可以有多条评论（1:N，dish_comments.dish_id）

- **reviews 表**与其他表的关系：
  - 一条评价属于一个用户（N:1，reviews.user_id -> users.id）
  - 一条评价属于一个店铺（N:1，reviews.shop_id -> shops.id）
  - 一条评价可以被多个用户点赞（1:N，review_likes.review_id）

## 5 功能模块设计

### 5.1 登录/注册模块

**功能描述**：用户通过微信授权一键登录，无需传统注册流程。

| 项目 | 说明 |
|------|------|
| 输入 | 用户点击"微信登录"按钮 |
| 处理 | 调用wx.login获取code，后端用code换取openid，查询或创建用户 |
| 输出 | 返回用户信息和登录态Token |
| 异常处理 | 网络异常时提示重试，授权拒绝时提示需要登录才能使用完整功能 |

### 5.2 首页推荐模块

**功能描述**：展示随机推荐、今日推荐、新店推荐等内容。

| 项目 | 说明 |
|------|------|
| 输入 | 用户进入首页 |
| 处理 | 1）从后端获取推荐店铺数据；2）渲染轮播图和列表；3）实现随机推荐动画 |
| 输出 | 展示推荐内容，支持点击进入店铺详情 |
| 交互设计 | "随便吃啥"功能采用卡片翻转动画，3秒内随机切换后定格推荐店铺 |

### 5.3 店铺浏览模块

**功能描述**：展示店铺列表，支持分类筛选、区域筛选、排序和搜索。

| 项目 | 说明 |
|------|------|
| 输入 | 用户滑动列表、选择筛选条件、输入搜索关键词 |
| 处理 | 1）监听筛选条件变化；2）发送请求获取筛选后的数据；3）渲染列表 |
| 输出 | 展示符合条件的店铺列表卡片 |
| 性能优化 | 列表使用虚拟滚动，图片懒加载，数据分页加载 |

### 5.4 店铺详情模块

**功能描述**：展示店铺完整信息，包括图片轮播、基本信息、评分分布、菜品列表和评价列表。

| 项目 | 说明 |
|------|------|
| 输入 | 用户点击店铺卡片 |
| 处理 | 1）加载店铺详情；2）加载评分分布数据（用于ECharts图表）；3）加载菜品列表；4）加载评价列表 |
| 输出 | 展示店铺详情页面，支持收藏、评价、分享操作 |
| 数据可视化 | 使用ECharts展示评分分布柱状图 |

### 5.5 评价互动模块

**功能描述**：用户可对店铺发表评价，对评价进行点赞和回复。

| 项目 | 说明 |
|------|------|
| 输入 | 用户填写评分、文字内容，可选上传图片 |
| 处理 | 1）校验评分和内容；2）上传图片到存储服务；3）提交评价数据；4）更新店铺评分统计 |
| 输出 | 评价成功后在评价列表顶部展示新评价 |
| 权限控制 | 仅登录用户可发表评价，游客引导登录 |

### 5.6 店铺/菜品上传模块

**功能描述**：用户可上传新店铺和菜品信息，丰富平台内容。

| 项目 | 说明 |
|------|------|
| 输入 | 用户填写表单信息，上传图片 |
| 处理 | 1）表单校验；2）图片压缩和上传；3）提交数据到后端；4）创建店铺/菜品记录 |
| 输出 | 上传成功后展示成功页面，新内容标注"新店"标签 |
| 激励设计 | 上传后给予用户正向反馈，鼓励UGC内容贡献 |

### 5.7 菜品互动模块

**功能描述**：用户可对菜品进行点赞、收藏、评论和分享。

| 项目 | 说明 |
|------|------|
| 输入 | 用户点击点赞、收藏、评论按钮 |
| 处理 | 1）调用对应接口；2）更新本地状态；3）刷新互动数据展示 |
| 输出 | 实时更新点赞数、收藏数、评论列表 |
| 交互设计 | 点赞时有微动效反馈，收藏后图标变化 |

### 5.8 个人中心模块

**功能描述**：展示用户个人信息、收藏列表、评价记录和上传记录。

| 项目 | 说明 |
|------|------|
| 输入 | 用户进入"我的"页面 |
| 处理 | 1）获取用户基本信息；2）统计收藏数量、评价数量、上传数量；3）加载对应列表数据 |
| 输出 | 展示个人中心页面，支持查看各类记录 |
| 扩展性 | 预留积分、等级等扩展功能入口 |

### 5.9 管理员后台模块

**功能描述**：管理员可管理店铺、评价、用户，查看数据统计。

| 项目 | 说明 |
|------|------|
| 输入 | 管理员登录后台，选择管理模块 |
| 处理 | 1）验证管理员权限；2）加载对应管理数据；3）支持增删改查操作 |
| 输出 | 展示管理列表，操作后实时刷新 |
| 安全措施 | 后台接口需要管理员Token验证，敏感操作记录日志 |

## 8 系统错误处理

### 8.1 出错信息管理

| 错误类型 | 错误提示 | 处理方式 |
|----------|---------|---------|
| 网络连接失败 | "网络开小差了，请检查网络后重试" | 提供重试按钮，本地缓存关键数据 |
| 服务器错误（500） | "服务器繁忙，请稍后重试" | 记录错误日志，自动重试机制 |
| 请求参数错误（400） | "请求参数有误" | 前端表单校验，避免非法参数 |
| 未授权（401） | "登录已过期，请重新登录" | 清除本地Token，引导重新登录 |
| 资源不存在（404） | "内容不存在或已被删除" | 返回上一页或首页 |
| 图片上传失败 | "图片上传失败，请重试" | 支持重新上传，压缩后重试 |
| 位置权限拒绝 | "需要位置权限以显示附近美食" | 引导用户开启权限，提供手动选择区域 |

### 8.2 故障预防与补救

| 预防措施 | 说明 |
|----------|------|
| 数据校验 | 前后端双重校验用户输入，防止SQL注入和XSS攻击 |
| 接口限流 | 对高频接口进行限流，防止恶意刷接口 |
| 图片压缩 | 前端上传前压缩图片，减少服务器压力和加载时间 |
| 数据备份 | 定期备份数据库，防止数据丢失 |
| 错误监控 | 接入错误监控平台，及时发现和修复问题 |

| 补救措施 | 说明 |
|----------|------|
| 数据回滚 | 数据库操作失败时自动回滚事务 |
| 降级策略 | 服务异常时展示缓存数据或默认页面 |
| 补偿机制 | 上传失败记录草稿，支持断点续传 |

### 8.3 系统维护设计

#### 8.3.1 编码规范化

- 前端代码遵循 Vue3 + UniApp 官方编码规范
- 后端代码遵循 ESLint + Prettier 规范
- 接口返回统一的数据格式：`{ code, message, data }`
- 数据库字段命名采用下划线命名法（snake_case）

#### 8.3.2 编码模块化

- 前端按页面和功能拆分组件，提高复用性
- 后端按模块拆分路由、控制器、服务层，遵循MVC架构
- 数据库操作封装为独立的Model层，使用Sequelize ORM
- 工具函数抽离为公共模块，如日期格式化、图片处理等

#### 8.3.3 始终收集反馈

- 用户评价功能本身就是反馈收集渠道
- 管理员后台可查看用户评价内容，了解用户需求
- 系统记录用户行为数据（浏览、搜索、收藏），用于优化推荐算法
- 预留意见反馈入口，收集用户使用建议
